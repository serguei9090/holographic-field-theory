# Findings Log 5: VRAM Footprint Scaling & Hierarchical Layer Paradigms
*Date: June 1, 2026*

This document provides a theoretical design overview comparing VRAM scaling behaviors between CHFT and traditional Transformer LLMs, and maps out implementation paradigms for adding "layers" of abstraction to CHFT without inheriting standard Transformer computational costs.

---

## 💾 1. VRAM Scaling: CHFT vs. Traditional LLM

For long-context execution (e.g. conversational histories), the primary bottleneck of standard Transformers is the **linear/quadratic growth of the KV-Cache** (Key-Value Cache) in VRAM. 

| Dimension / Context Scale | Standard LLM (Transformer) | CHFT (VSA + Hopfield) |
| :--- | :--- | :--- |
| **Small Context (e.g. 8 tokens)** | **1.7 GB - 2.0 GB** (Slightly lower due to simple real-valued embedding tables). | **3.5 GB - 4.0 GB** (Higher base footprint due to `complex64` key matrix expansion). |
| **Long Context (e.g. 100,000+ tokens)** | **Memory Explosion (10 GB - 30+ GB)**. VRAM scales linearly $O(N)$ with history length to store every token's keys and values. | **Constant Memory (~4.0 GB)**. The history is bound and bundled into a single vector ($\psi$). VRAM remains flat. |

### VSA Context Footprint
For a dimension $D = 16,384$, the context vector $\psi$ holds the entire sequence history in a single complex array:
$$\text{Memory Size} = 16,384 \times 8\text{ bytes} \approx 131\text{ KB}$$
This memory size remains **exactly 131 KB** whether the context holds 8 tokens or 1,000,000 tokens, bypassing the memory bandwidth bottleneck of standard LLMs.

## 🧱 2. Stacking Hierarchical Layers in CHFT

In traditional Transformers, "depth" is added by stacking self-attention and MLP blocks. In a VSA/Hopfield architecture, stacking layers introduces symbolic abstraction levels:

```
                  [Concept Vector (Doc Level)]
                              ▲
                              │ Layer 3: Paragraph Bundling
                              ▼
                 [Paragraph Vector (Block Level)]
                              ▲
                              │ Layer 2: Sentence Bundling
                              ▼
                  [Sentence Vector (Phrase Level)]
                              ▲
                              │ Layer 1: Token Binding
                              ▼
                     [Vocabulary Tokens]
```

### Type A: Attractor Refinement Layers (Deep Reasoning)
* **Design**: Increasing the iterative attractor lookup cycles ($N$ steps) inside `ModernHopfieldMemory.refine_query` or training separate projection layers to guide the query settling path.
* **Model Size Impact**: **0 parameters** (if utilizing algorithmic iterations on existing keys).
* **VRAM Impact**: **0 MB** (purely reuse of GPU memory buffers).
* **Compute Impact**: Minimal increase in training/inference latency.

### Type B: Hierarchical Context Layers (Sentence/Paragraph Levels)
* **Design**: Using VSA algebraic operators (rotations/permutations) to recursively bind words into sentence vectors, and sentence vectors into paragraph vectors.
* **Model Size Impact**: 
  * **0 parameters** if using fixed algebraic projection rotations.
  * **~250M parameters (~1 GB weights)** if using trainable feed-forward transition matrices.
* **VRAM Impact**: **Virtually Free** (~131 KB per abstraction level to store active vectors).

### Type C: Translation/Feed-Forward Layers
* **Design**: Inserting a dense projection layer between Hopfield retrieval outputs and next-token prediction to resolve clean categorical labels and enforce grammatical rules.
* **Model Size Impact**: Moderate increase (~268M parameters or ~1 GB of weights).
* **VRAM Impact**: Increases by **~1 GB** to store the translation layer weights in GPU memory.
